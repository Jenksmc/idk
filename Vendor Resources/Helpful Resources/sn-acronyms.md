## Frequently Used Acronyms Dictionary

The following is a list of the most widely used acronyms used by teams working on AGC Service Notifications: 

>[!TIP]
>Operators can also search the [Microsoft Corporate Glossary](https://microsoft.sharepoint.com/SitePages/Glossary.aspx) to find definitions of other acronyms, words, and phrases. 

| ACRONYM | TERM                | DEFINITION         |
| ------- | ------------------- | ------------------- |
| **ACE** | Advanced Cloud Engineering | A Microsoft program with a mission to ensure world-class support customer experiences for Azure customers across delivery channels​. ACE team members often act as technical consultants on critical Outages. |
| **ACM** | Azure Communications Manager | An internal Microsoft system that publishes service health communications to Azure customers. |
| **ADO** | Azure DevOps | A collection of Microsoft services and technologies for agile planning, continuous integration, continuous delivery, and monitoring of applications. |
| **AGC** | AirGap Cloud | Cloud environments that are completely separated from the worldwide commercial environments. |
| **BRAIN** | Blackout Resilient Azure Intelligent Network | An internal Microsoft system that leverages automation and intelligence to improve outage management for Azure. |
| **C+AI** | Cloud + AI platform | A Microsoft business unit responsible for the company’s computing fabric and artificial intelligence platform. | 
| **CCB** | Crisis Control Bridge | A communications channel for all service and field teams that are not key to restoration of the platform during a major Live Site Incident. NOTE: This was formerly called the "Partner Control Bridge (PCB)". |
| **CEM** | Crisis Escalation Manager | Individuals who assist internal and external customers and field and support personnel and who manage partner channel status updates during live site issues. |
| **CLI** | Command-Line Interface | Text-based interfaces used to interact with software and operating systems. |
| **CSM** | Customer Support Manager | Individuals who oversee teams that provide support to customers. | 
| **CXP** | (Azure) Customer Experience | A Microsoft business unit that provides customized build and deployment, engineering support, outage and security incident management, and customer research and feedback loops for cloud customers. | 
| **DRI** | Designated Responsible Individual | Individuals responsible for live site incident management and mitigation. Also known as "Directly Responsible Individual". | 
| **DRS** | Detailed Resolution Statement | The former name of the AzComms customer message that describes in detail the issue and how it was mitigated. Now called the Mitigation Statement. |
| **EIM** | Executive Incident Manager | Engineering managers responsible for ensuring that investigations, mitigations, and service restorations for complex and severe incidents occur in a timely manner. | 
| **EITC** | Enough Information To Communicate | The minimum amount of information needed to send the initial communication. | 
| **FDP** | First Detailed Post | AzComms message sent to customers that includes a more detailed description of the issue (called the "Impact Statement"). | 
| **FTE** | Full Time Employee | Microsoft full-time employees involved with the AGC AzComms team may include the EIM, DRI, Engineering Manager, and CSM. | 
| **IcM** | Incident Management System | An internal Microsoft incident management system that provides tools for managing live site and on-call rotations for Microsoft services. |
| **JIT**  | Just-In-Time | An internal Microsoft service that provides temporary elevated access to engineers to debug production issues or support customer cases. |
| **KQL** | Kusto Query Language | A query language used with Microsoft services. |
| **LSI** | Live Site Incident | Events or sets of events that cause interruption, disruption, loss, or damage to customer-facing systems. | 
| **MFA** | Multi-Factor Authentication | An authentication method that requires users to present two or more forms of evidence of identity. |
| **PCB** | Partner Control Bridge | A communications channel for all service and field teams that are not key to restoration of the platform during a major Live Site Incident. NOTE: This is now called the "Crisis Control Bridge (CCB)". |
| **PIR** | Post Incident Review | AzComms message sent to customers documenting the findings of the Outage root-cause analysis. | 
| **PPE** | Pre Production Environment | Deployment environments used to validate software and content under conditions similar to a production environment. |
| **RA** | Request Assistance | Process by which Service Teams request customer communications assistance from the AGC AzComms team. Service Teams RA by clicking on a button in IcM. |
| **RCA** | Root Cause Analysis | A process of analyzing cause and effect relationships of undesirable events to identify actions to prevent recurrence. |
| **SAW** | Secure Administration Workstation | A secure workstation that is used for employees with access to production environments. |
| **SLA** | Service Level Agreement | Agreements between service providers and clients that include service details, standards, metrics, and courses of action if requirements are not met. | 
| **SME** | Subject Matter Expert | Individuals with highly specialized knowledge or expertise in specific areas. | 
| **SRC** | Simple Resolved Comms | AzComms message sent to customers to quickly notify them that an Outage has been mitigated. | 
| **ST** | Service Team  | The Azure engineering team making the change to the customer-facing AGC Service. | 
| **ST PM** | Service Team Project Manager | Program manager responsible for planning the PDLC of a service; usually the SN requestor. | 
| **SN** | Service Notification | AzComms message sent to customers to notify them of impactful and non-impactful upcoming scheduled platform and Service changes. | 
| **TLA**  | Three Letter Acronym | Redundant. |
| **TSG** | Troubleshooting Guide | Documents and content that provide instructions for diagnosing and solving common product issues. (Sometimes also referred to as "Technical Service Guides".)| 
| **TTA** | Time to Acknowledge | A metric that measures the time between when an incident is assigned to a team, or request assistance is initiated, and when the designated response individual acknowledges the incident or request. |
| **TTD** | Time To Deliver | A metric that measures the time elapsed after a request has been submitted until it has been completed - or delivered. (Also may be used by some teams as "Time to Detect".) |
| **TTM** | Time To Mitigate | A metric that measures the difference between the time an issue occurs and the time customer impact is mitigated and customer SLA is reestablished. |
| **TTP** | Time to Post | A metric that measures the difference between the time an issue occurs and the time the customer communication is published in ACM. |
| **VM** | Virtual Machine | A software-based emulation of a physical computer which operates like a physical computer with its own CPU, memory, storage, and network interface, but it exists only as code within a physical host computer. |
| **USNat** | TERM | A "High Side" airgapped cloud environment used at Microsoft. | 
| **USSec** | TERM | A "High Side" airgapped cloud environment used at Microsoft. |
